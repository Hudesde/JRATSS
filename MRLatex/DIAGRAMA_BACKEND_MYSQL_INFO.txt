4. CONEXIÓN A MySQL
==========================================================

¿Usa mysql2/promise? NO
Usa: mysql2 (versión callback convertida a promesas con util.promisify)

Archivo de conexión: backend/src/dataBase.ts

Nombre de la clase/pool: pool (default export)

Código:
```typescript
import mysql from 'mysql2';
import dotenv from 'dotenv';

dotenv.config();

const pool = mysql.createPool({
  host: process.env.DB_HOST || 'localhost',
  user: process.env.DB_USER || 'root',
  password: process.env.DB_PASSWORD || '',
  database: process.env.DB_NAME || 'charlestherapy',
  waitForConnections: true,
  connectionLimit: 10,
  queueLimit: 0
});

export default pool;
```

Uso en controladores:
```typescript
import pool from '../dataBase';

// Ejemplo:
const [rows] = await pool.execute('SELECT * FROM usuarios WHERE id_usuario = ?', [id]);
```

5. ESTRUCTURA DE LA BASE DE DATOS (TABLAS)
==========================================================

Base de datos: charlestherapy

Tablas principales:

1. usuarios
   - id_usuario (INT, PK, AUTO_INCREMENT)
   - usuario (VARCHAR)
   - correo (VARCHAR)
   - password (VARCHAR)
   - rol (ENUM: 'Paciente', 'Especialista')
   - bigFive (TINYINT: 0 = no completado, 1 = completado)
   - cedula (VARCHAR, nullable)
   - especialidad (VARCHAR, nullable)

2. bigfive
   - id_resultado (INT, PK, AUTO_INCREMENT)
   - id_usuario (INT, FK -> usuarios)
   - neuroticismo (DECIMAL)
   - extraversion (DECIMAL)
   - apertura (DECIMAL)
   - amabilidad (DECIMAL)
   - responsabilidad (DECIMAL)
   - fecha (DATETIME, DEFAULT CURRENT_TIMESTAMP)

3. cita
   - id_cita (INT, PK, AUTO_INCREMENT)
   - id_paciente (INT, FK -> usuarios)
   - id_especialista (INT, FK -> usuarios)
   - fecha (DATE)
   - hora (TIME)
   - motivo (TEXT)
   - estado (ENUM: 'pendiente', 'confirmada', 'cancelada')
   - fecha_creacion (DATETIME, DEFAULT CURRENT_TIMESTAMP)

4. mensajes
   (estructura por confirmar, posiblemente para chat interno)

5. observaciones
   - id_observacion (INT, PK, AUTO_INCREMENT)
   - id_usuario (INT, FK -> usuarios)
   - observacion (TEXT)
   - fecha (DATETIME, DEFAULT CURRENT_TIMESTAMP)

6. sesiones
   - id_sesion (INT, PK, AUTO_INCREMENT)
   - id_usuario (INT, FK -> usuarios)
   - contenido (TEXT: JSON stringificado con mensajes del chat)
   - fecha_creacion (DATETIME, DEFAULT CURRENT_TIMESTAMP)

7. resumen (tabla opcional para resúmenes diarios/semanales)
   - id_resumen (INT, PK, AUTO_INCREMENT)
   - id_usuario (INT, FK -> usuarios)
   - fecha (DATE)
   - contenido (TEXT)
   - resumen (TEXT)
   - fecha_creacion (DATETIME, DEFAULT CURRENT_TIMESTAMP)

7. FLUJO DE DATOS (EJEMPLO: CREAR SESIÓN DE CHAT)
==========================================================

1. Frontend hace POST a /api/sesiones
   Body: { id_usuario: 11, contenido: "[{role:'user', content:'Hola'}]" }

2. Express enruta a sesionChatRoutes

3. sesionChatRoutes llama a sesionChatController.crearSesion

4. crearSesion valida datos y ejecuta:
   await pool.execute(
     'INSERT INTO sesiones (id_usuario, contenido) VALUES (?, ?)',
     [id_usuario, contenido]
   )

5. MySQL inserta el registro y devuelve insertId

6. Controlador responde con JSON:
   { message: 'Sesión creada', id_sesion: 123 }

7. Frontend recibe la respuesta y guarda id_sesion

==========================================================

Archivo principal: backend/src/index.ts

router.put('/:id_sesion', actualizarSesion);

export default router;

11. COMPILACIÓN Y EJECUCIÓN
==========================================================

Scripts disponibles:
- npm run dev: Inicia con nodemon (desarrollo)
- npm run build: Compila TS a JS (carpeta build/)
- npm start: Ejecuta build/index.js

Proceso de compilación:
1. tsconfig.json define opciones de compilación
2. tsc compila .ts a .js en src/build/
3. nodemon ejecuta src/build/index.js
4. En producción, ejecutar directamente node src/build/index.js
